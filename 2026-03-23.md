**GENERICS**


**What Are Generics?**

Generics allow you to define a class, function, or interface with a type parameter instead of a fixed type.

**Example:**
          
          class Box<T>(val value: T)
          //T is a type parameter
          //You can use it with any type:
          fun main(){
          val intBox = Box(10)        // T = Int
          val stringBox = Box("Hi")   // T = String
          }

          
 **Why Use Generics?**

Without generics:

You use Any → unsafe casting
Runtime errors possible

With generics:

Type safety 
No casting needed 
Reusable code 


**Generic Functions**

            //You can also use generics in functions:
            
            fun <T> printValue(value: T) {
                println(value)
            }
            
            fun main(){
            
            printValue(10)
            printValue("Hello")
            }


            
 **Multiple Type Parameters**

 
      class PairBox<K, V>(val key: K, val value: V)

      
**Generic Constraints (Upper Bounds)**

You can restrict types:

        fun <T : Number> process(value: T) {
            println(value.toDouble())
        }

Only subclasses of Number allowed.



**Where Clause (Multiple Constraints)**

        
        fun <T> example(value: T) 
            where T : Number, T : Comparable<T> {
            println(value)
        }

        
Example:

                              interface Printable {
                                  fun print()
                              }
                              
                              open class Document
                              
                              class Report : Document(), Printable {
                                  override fun print() {
                                      println("Printing Report")
                                  }
                              }
                              fun <T> process(item: T) where T : Document, T : Printable {
                                  item.print()
                              }
                              
                              fun main(){
                                  val report = Report()
                              	process(report) // ✅ Works
                              }


**VARIANCE**


Variance defines how generic types behave when inheritance is involved.



We know:

String is a subtype of Any

But:

List<String> ❌ List<Any>

Why?-----Because generics are invariant by default in Kotlin.

**1. Invariance (Default Behavior)**


        class Box<T>(val value: T)


Box<String> is NOT a subtype of Box<Any>

 Even though String → Any

Why?

Because allowing it could break type safety.

**Covariance (out) → Producer**

Meaning:“This class only PRODUCES (returns) T”

          class Box<out T>(val value: T)

Now:

Box<String> ✅ Box<Any>

✔ Safe because you're only reading

Example:

        fun print(box: Box<Any>) {
            println(box.value)
        }
        fun main(){
        val stringBox = Box("Hello")
        print(stringBox) // Works!
        }


Restriction:

You CANNOT write into it:

          class Box<out T> {
              fun add(value: T) {} // ERROR
          }

 Because type safety would break.

**Contravariance (in) → Consumer**

Meaning:

“This class only CONSUMES (accepts) T”

          class Box<in T> {
              fun add(value: T) {}
          }

Now:

Box<Any> ✅ Box<String>

✔ Safe because you're only writing

Example:
            fun fill(box: Box<String>) {
                box.add("Hello")
            }
            fun main(){
            val anyBox = Box<Any>()
            fill(anyBox) // Works!
            }

            
Restriction:

You CANNOT read safely:

val item: T ❌ NOT ALLOWED

Because actual type could be broader.


| Type          | Keyword | Direction           | Use     |
| ------------- | ------- | ------------------- | ------- |
| Invariant     | (none)  |  No relation       | Default |
| Covariant     | `out`   | Subtype → Supertype | Read    |
| Contravariant | `in`    | Supertype → Subtype | Write   |



**Kotlin Built-in Examples**


**Covariant:**

        interface List<out T>

✔ You can read from a list safely

**Contravariant:**

          interface Comparator<in T>

✔ It consumes values to compare


Note:

Without variance, this would be unsafe:

          val list: MutableList<String> = mutableListOf("A")
          
          val anyList: MutableList<Any> = list //error

        anyList.add(123) // Now list has Int →  crash later



**Star Projection**

* means “unknown generic type”, but handled safely

Example

        
        val list: List<*> = listOf("A", "B", "C")

This means:

    The list has some type
    
    But we don’t know what it is

It could be:

    List<String>
    
    List<Int>
    
    List<Anything>


**Why Do We Need It?**

Imagine writing a function that works with any generic list:

            fun printList(list: List<*>) {
                for (item in list) {
                    println(item)
                }
            }

This works for:

List<String>

List<Int>

List<Double>


**Key Behavior**

You can READ
      
      val item = list[0]

 Type of item = Any?

 You CANNOT WRITE
 
        val list: MutableList<*> = mutableListOf<String>()
        list.add("Hello") ❌ ERROR

Only allowed:

      list.add(null) ✅

**Why Can’t You Write?**

Because Kotlin doesn’t know the actual type.

 Example:

 
        fun main(){
        val list: MutableList<*> = mutableListOf<Int>()

          //If Kotlin allowed:
          
          list.add("Hello") //  would break type safety
          }


**Star Projection vs Other Types**

**List<Any>**

val list: List<Any>

You KNOW it's Any

Can hold mixed types

**List<*>**

val list: List<*>

You DON'T know type

Could be strictly List<String>

BIG difference:

List<Any> = flexible

List<*> = unknown but safe

**Real-Life Analogy**

Boxes:

Box<Any> → you can put anything inside

Box<*> → sealed box → you don’t know what’s inside → don’t touch




**When to Use Star Projection**

Use * when:

You don’t care about the type

You only need to read

You want generic-safe APIs

Example:

        fun printSize(list: List<*>) {
            println(list.size)
        }

**Type Erasure**


Type erasure means generic type information is removed at runtime

Example

val list: List<String> = listOf("A", "B")

At compile time:

Kotlin knows → List<String>

At runtime (JVM sees):

List

// The <String> is erased

**Why this is a problem**

You cannot check generic types at runtime:

            if (list is List<String>) ❌ ERROR

Because JVM only sees:

      if (list is List) 


Another Example


            fun <T> check(value: Any) {
                if (value is T) ❌ ERROR
            }

Why?

Because T is erased → JVM doesn’t know what T is

**Why Type Erasure Exists**

JVM was created before generics existed

Generics were added later → implemented using erasure

Ensures backward compatibility

Note:

Generics are only for compile-time safety, not runtime

 **Reified (Solution to Type Erasure)**
 

reified allows you to access generic type at runtime


**Without reified**

            fun <T> isType(value: Any): Boolean {
                return value is T // ❌ NOT allowed
            }

            
**With reified**

inline fun <reified T> isType(value: Any): Boolean {
    return value is T // ✅ works
}


**Why It Works**

Because of inline

What inline does:

Copies function code at call site

Example:

      isType<String>("Hello")

Gets converted to:

"value is String"

Now JVM knows the type

**Important Rule**

reified only works with:

      inline fun <reified T>


 **Real Use Case**
 
JSON parsing example
        
        inline fun <reified T> parse(json: String): T {
            return Gson().fromJson(json, T::class.java)
        }

Without reified, this wouldn’t work.

| Feature                   | Type Erasure     | Reified             |
| ------------------------- | ---------------- | ------------------- |
| Type at runtime           | ❌ Removed        | ✅ Available         |
| Works in normal functions | ✅                | ❌                   |
| Requires `inline`         | ❌                | ✅                   |
| Use case                  | General generics | Runtime type checks |



**Retrofit**

Retrofit in Kotlin is a popular HTTP client library used to make API calls (usually REST APIs) in Android apps. It simplifies networking by turning your API endpoints into Kotlin interfaces.


**Step 1: Add Dependencies**



You need to include the required libraries in your project so that you can use Retrofit.

Why it is needed:----Retrofit handles HTTP requests (GET, POST, etc.)

The Gson converter converts JSON responses into Kotlin objects


        implementation 'com.squareup.retrofit2:retrofit:2.9.0'
        implementation 'com.squareup.retrofit2:converter-gson:2.9.0'




This step enables networking and JSON parsing in your app.

**Step 2: Add Internet Permission**




You must allow your Android app to access the internet.

Why it is needed----Without this permission, network requests will not work, even if your code is correct.

            
            <uses-permission android:name="android.permission.INTERNET"/>

Where to add

Inside the AndroidManifest.xml file.



This step gives your app permission to make API calls.

**Step 3: Create Data Model (Response Class)**



You create a Kotlin data class that matches the structure of the JSON response from the API.

**Example JSON**

            {
              "userId": 1,
              "id": 1,
              "title": "Hello",
              "body": "World"
            }

            
**Kotlin Data Class**

            data class Post(
                val userId: Int,
                val id: Int,
                val title: String,
                val body: String
            )

            
Why it is needed---Retrofit uses the converter to automatically map JSON data into this class.



This step defines how the API response will be stored in your app.

**Step 4: Create API Interface**



You define the API endpoints using a Kotlin interface.


            import retrofit2.http.GET
            
            interface ApiService {
            
                @GET("posts")
                suspend fun getPosts(): List<Post>
            }

            

@GET("posts") specifies the endpoint

suspend allows the function to run asynchronously using coroutines

List<Post> is the expected response type

Why it is needed---Retrofit reads this interface and generates the actual implementation for making network calls.



//This step describes what request to send and what response to expect.
